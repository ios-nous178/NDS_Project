# ds-fix — 목업 피드백 → 디자인시스템 자율 수정 루프

> **Codex skill (생성됨).** SSOT 는 `.claude/skills/ds-fix/SKILL.md` — 직접 수정 금지, SSOT 수정 후 `pnpm sync:skills`. 명시 호출 `/skills` → `ds-fix` 또는 `$ds-fix <피드백 텍스트>`. 컴포넌트 수정은 ds-component 플로우 재사용.

목업은 **probe** 다. 목업을 만들거나 리뷰하다 나온 피드백은 대개 **DS 의 공백·버그·가이드 누락**을 드러낸다.
이 스킬은 그 피드백을 받아 **목업이 아니라 DS 를** 고친다(컴포넌트/토큰/패턴/검증룰). 피드백이 여러 표면에 걸치므로 **상위 디스패처**로 동작하며, 컴포넌트 수정은 `/ds-component` 플로우를 재사용한다.

`/ds-component` 와 독립이다 — 알려진 컴포넌트 하나를 손볼 땐 `/ds-component` 를 직접 쓰고, "피드백 덩어리에서 알아서 고쳐줘" 일 땐 이 스킬을 쓴다.

## 입력 (자유 텍스트 우선)

- **1차**: 사용자가 **붙여넣은 자유 텍스트** — 다른 AI 의 답변/크리틱, QA 코멘트, 이슈. (정제된 폼이 아니다. 이게 보통의 입력이다.)
- **옵션**: 대상 목업 파일/화면(rel 경로), 브랜드.
- **보조**: `.ds-feedback-log.jsonl` 의 해당 screen `revision-request` 항목, 그리고 그 목업에 `validate_html_mockup`/`score_mockup_quality` 를 돌린 현재 위반·점수.

## 카디널 룰

1. **목업이 아니라 DS 를 고친다.** "이 목업이 이상해" → 근본 원인이 DS(컴포넌트/토큰/가이드/룰)면 거기를 고친다. 목업 한 파일만 땜질하고 끝내지 않는다.
2. **DS 공백은 hack 하지 말고 escalate.** 없는 컴포넌트/토큰을 raw hex·custom div 로 때우지 않는다 — 신설이 정당하면 신설(=`/ds-component`/토큰 절차), 디자인 판단이 필요하면 사람에게.
3. **오용은 재발 방지로 닫는다.** 피드백이 "작성자가 DS 를 잘못 씀"이면, 고칠 DS = **검증룰/가이드를 강화**해 같은 실수가 다시 안 나게 한다(+ 필요 시 목업도 고쳐야 함을 보고).
4. **토큰-퍼스트·3면 미러·외부전파** — `/ds-component` 카디널 룰 그대로 따른다(매핑 안 되는 hex 는 flag, react/styles/html lockstep, 카탈로그/가이드/changeset).
5. **자율이되 안전.** max iteration·무진척 감지로 thrashing 을 막고, 막히면 멈춰서 보고한다.
6. **고치기 전에 먼저 합의한다 (★ 사용자 기본).** 트리아지(1단계)가 끝나면 **곧바로 수정에 들어가지 말고**, 먼저 사용자에게 **① 문제 정의(원문→근본원인) → ② 해결방안 제안(어디를 어떻게)** 을 제시하고 **착수 승인을 받는다**(1.5단계). 이 확인은 **루프 초반(트리아지 직후)** 에 하지, 다 고친 뒤 보고에서 하지 않는다. 사용자가 "그냥 다 해줘"라고 미리 위임했거나 자명한 1건짜리면 게이트를 생략할 수 있다.

## 루프

### 0. 파싱 & 컨텍스트

- 붙여넣은 텍스트를 **개별 이슈로 분해**(한 덩어리에 여러 건이 섞여 있다). 대상 목업/화면이 명시 안 됐으면 물어보거나 텍스트에서 추론.
- 대상 목업이 있으면 `validate_html_mockup` + `score_mockup_quality` 로 객관 신호도 수집해 피드백과 합친다.

### 1. 트리아지 (각 이슈를 분류) ★ 핵심

- **DS 버그/공백** — 컴포넌트가 시안과 다르다 / 컴포넌트·토큰이 없다 / 가이드가 틀렸다 / 검증이 못 잡았다 → **DS 수정**.
- **목업 오용** — DS 는 맞는데 작성자가 잘못 씀 → **가이드/검증룰 강화**(재발 방지) + 목업 수정 필요 보고.
- **디자인 판단 / 모호 / 시안 부재** — 토큰에 없는 값, 브랜드 정책, 픽셀 결정 등 → **escalate**(사람). 지어내지 않는다.

### 1.5 제안 & 승인 게이트 (사용자 확인 — 초반에) ★

트리아지가 끝나면 **수정 착수 전에** 사용자에게 다음을 한 번에 제시하고 승인을 받는다:

- **문제 정의** — 이슈별로 `원문 한 줄 → 무엇이 잘못됐나 → 근본 원인이 DS 어디인가(파일/토큰/룰)`.
- **분류** — 버그/공백 · 오용 · 디자인판단(escalate) 중 무엇인지.
- **해결방안 제안** — 표면별로 어떻게 고칠지(컴포넌트 3면/토큰/패턴/룰). 외부전파(가이드·changeset) 포함 여부.
- **확인 질문** — 어떤 항목을 고칠지 / escalate 항목의 방향. `AskUserQuestion` 으로 묻는다(애매·디자인결정 위주).

승인 후에만 2단계(라우팅·수정)로 간다. 무진척·디자인결정 필요로 다시 막히면 같은 방식으로 재합의한다.

### 2. 라우팅 (표면별로 고친다)

| 이슈 유형                     | 손대는 곳                                                                                                         |
| ----------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| 컴포넌트가 시안과 다르다/없다 | **`/ds-component` 플로우 재사용** — react/styles/html 3면 + 스토리 + AllComponents + COMPONENT_GUIDES + changeset |
| 색/크기 토큰 문제             | `packages/tokens/src/**` (+ 브랜드) + `DESIGN.md` + `pnpm build --filter @nudge-design/tokens`                    |
| 합성/사용 규칙(패턴)          | `guides.ts` `PATTERN_GUIDES` + (필요 시) `html-validator.ts` 검증룰 + 예시                                        |
| 검증이 못 잡음                | `packages/mockup-core/src/tools/html-validator.ts` 룰 추가 + 테스트 + (catalog-config 컨텍스트)                   |
| 가이드만 틀림/누락            | `guides.ts` `COMPONENT_GUIDES`/`PATTERN_GUIDES` 본문                                                              |

### 3. 검증 (이슈가 실제로 닫혔는지)

- `/ds-component` 검증 게이트와 동일: 패키지별 `tsc` + `vitest`(react/html) + `node --test`(mockup-core) + **정합 게이트**(빌드된 브랜드 CSS ↔ 시안 치수) + 예시 `validate_html_mockup`.
- **이슈 재현 시나리오로 확인** — 그 피드백을 유발한 입력(목업 조각/오용 패턴)에 다시 돌려 위반/점수/외형이 실제로 해소됐는지 본다. (검증룰을 추가했으면, 그 룰이 해당 오용을 이제 잡는지 테스트로 고정.)

### 4. 종료 판정

- ✅ **완료**: 모든 이슈가 (a) 고쳐지고 검증 통과 **또는** (b) escalate 로 분류됨. + 건드린 패키지 tsc/테스트 green.
- 🛑 **중단·보고**:
  - max iteration(기본 5) 초과
  - **무진척** — 2회 연속 같은 이슈/위반이 안 줄면 thrashing → 멈춤(원인 분석을 사람에게)
  - **사람/디자인 결정 필요** — escalate 항목이 남음

### 5. 보고 (커밋 X)

- **이슈별 처리표**: 원문 한 줄 → 분류(버그/오용/escalate) → 손댄 DS 표면 → 어떻게 고쳤나 / 왜 escalate.
- **before → after**: 위반 수·score verdict·dsRatio(있으면).
- **잔여**: 사람 결정 필요 항목 + 권장 방향.
- 마무리는 `/ds-component` 와 동일 — 코드 컴포넌트를 손댔으면 `pnpm --filter storybook dev` 띄워 시각 확인 안내. **자동 커밋하지 않는다.**

## 흔한 함정 (이 레포 회고)

- 피드백을 목업 한 파일에서만 고치고 DS 를 안 고침 → 다른 목업에서 같은 문제 재발. 근본은 DS.
- 시안 hex 가 토큰에 없다고 새 값을 지어냄(예: `#60be34`, `#111`) → 금지. 토큰 매핑 실패는 escalate.
- 오용을 고치면서 검증룰을 안 박음 → 재발. "오용"의 마무리는 **룰/가이드**.
- 동시 세션 충돌: 손대기 전 `git status` 로 대상 파일이 이미 dirty(다른 세션)면 손대지 말 것.
- 외부전파 누락(AllComponents/가이드/changeset)이 "고쳤는데 외부엔 안 나감"의 원인.
- **승인 없이 바로 고치기** → 사용자는 착수 전 문제정의+해결방안 합의를 원한다(카디널 6·1.5). 트리아지하면 먼저 제시하고 묻는다.

## 안 하는 것

- 목업 자체를 산출(→ 목업 플로우). 자동 커밋/푸시·릴리즈(MCPB)는 별도.
- 시안/근거 없이 디자인 결정(색·간격·정책)을 임의 확정 — escalate.
