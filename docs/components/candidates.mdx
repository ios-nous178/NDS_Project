# 디자인시스템 추출 후보 목록

`NudgeEAPHomePage`와 `NudgeEAPWebView`를 함께 읽어보며, 반복적으로 등장하고 디자인시스템으로 승격했을 때 관리 효율이 커질 만한 컴포넌트를 정리했습니다.

판단 기준은 아래 4가지입니다.

- 두 프로젝트에서 비슷한 UI 패턴이 반복되는가
- 도메인 로직보다 UI 규칙과 상태 관리가 더 중요한가
- 접근성, 상태, 스타일 확장 규칙을 공통으로 관리할 가치가 큰가
- Storybook과 Figma 문서로 설명하기 좋은 구조인가

## 1. Search Input / Search Bar

가장 먼저 공통화해볼 만한 후보입니다. 검색창은 이미 프로젝트 전반에서 비슷한 구조로 반복되고 있고, placeholder, clear 버튼, 검색 아이콘 버튼, 둥근 field 스타일이 거의 같은 톤으로 쓰이고 있습니다.

대표 구현:

- `NudgeEAPHomePage/src/app/(primary)/letter/components/LetterSearchInput.tsx`
- `NudgeEAPHomePage/src/app/(primary)/find-id-pw/components/SearchBar.tsx`
- `NudgeEAPHomePage/src/app/(primary)/service/search/partner/components/PartnerSearchInput.tsx`
- `NudgeEAPHomePage/src/app/(primary)/components/EapHomeSearchBox.tsx`
- `NudgeEAPWebView/src/app.feature/weekly-letter/component/WeeklyLetterSearchBar.tsx`

추천:

- 현재 `Input`의 compound API를 기반으로 `search` variant를 정식 지원
- 또는 `SearchInput`을 `Input` 위의 레시피 컴포넌트로 분리
- clear/search action slot, mobile width, icon button spacing을 토큰화

## 2. Tabs / Sliding Tabs

웹뷰 쪽에서 특히 반복이 많고, 변형도 이미 체계가 있습니다. 현재는 구현이 여러 파일로 나뉘어 있지만 디자인시스템 관점에서는 `Tabs` 하나와 `variant` 조합으로 정리하기 좋은 상태입니다.

대표 구현:

- `NudgeEAPWebView/src/app.component/slidingTab/SlideTab.tsx`
- `NudgeEAPWebView/src/app.component/slidingTab/StyledLineTab.tsx`
- `NudgeEAPWebView/src/app.component/slidingTab/StyledPillTab.tsx`
- `NudgeEAPWebView/src/app.component/slidingTab/StyledSquareTab.tsx`
- `NudgeEAPWebView/src/app.component/slidingTab/StyledCircleTab.tsx`

추천:

- `Tabs.Root`, `Tabs.List`, `Tabs.Trigger`, `Tabs.Indicator`, `Tabs.Panel` 구조로 정리
- `line`, `pill`, `square`, `circle` variant 제공
- 스크롤 가능 탭과 indicator 애니메이션을 기본 기능으로 포함

## 3. Bottom Sheet

모바일 UX에서 자주 쓰이는 핵심 오버레이입니다. 현재 `Modal`은 이미 있으니, 그 다음 공통 컴포넌트로 `BottomSheet`를 올리기 좋은 상태입니다.

대표 구현:

- `NudgeEAPWebView/src/app.component/bottomSheet/BottomSheet.tsx`
- `NudgeEAPWebView/src/app.component/bottomSheet/BottomSheetContent.tsx`
- `NudgeEAPWebView/src/app.component/modal/ModalSheetBottom.tsx`
- `NudgeEAPHomePage/src/app/(primary)/service/components/CounselingAppDownloadModal.tsx`

추천:

- `Modal`과 같은 접근성 규칙을 공유하되, 바닥 정렬과 드래그 핸들 스타일을 포함
- header, body, footer slot 제공
- full / fit-content height variant 지원

## 4. Select / Filter Select

지금은 브라우저 기본 select와 커스텀 필터 선택 UI가 섞여 있습니다. 검색/필터/가입 폼을 생각하면 공통 선택 컴포넌트가 필요한 상태입니다.

대표 구현:

- `NudgeEAPWebView/src/app.feature/counseling/component/FilterSelect.tsx`
- `NudgeEAPHomePage/src/app/(primary)/sign-up/components/SelectItem.tsx`
- `NudgeEAPHomePage/src/app/(primary)/sign-up/components/CounselTypeSelect.tsx`

추천:

- 기본형 `Select`와 검색/필터용 `SelectTrigger + Listbox` 구조를 함께 설계
- 모바일 바텀시트 선택과 데스크톱 드롭다운 선택을 같은 인터페이스로 묶기

## 5. Checkbox / Radio / Choice Controls

현재는 개별 파일로 구현되어 있고, label, helper text, error, group spacing 규칙이 아직 통일돼 있지 않습니다.

대표 구현:

- `NudgeEAPHomePage/src/app/(primary)/components/Checkbox.tsx`
- `NudgeEAPHomePage/src/app/(primary)/components/Radio.tsx`
- `NudgeEAPWebView/src/app.feature/feed/page/feedDetail/Feed.Report.tsx`

추천:

- `Checkbox`, `Radio`, `ChoiceGroup`로 나누고 field label/helper/error를 공통 규칙으로 정리
- 단일 선택과 다중 선택의 spacing, icon, checked state를 토큰화

## 6. Toast

웹뷰에는 이미 토스트 인프라가 있고, 홈에서도 사용자 피드백 컴포넌트 수요가 분명합니다. 성공/실패/정보 메시지 규칙을 공통화하기 좋습니다.

대표 구현:

- `NudgeEAPWebView/src/app.component/toast/ToastDomAlert.tsx`
- `NudgeEAPWebView/src/app.component/toast/toast.controller.ts`

추천:

- `ToastProvider`, `ToastViewport`, `Toast` 구조로 설계
- success, error, info variant와 action button 포함 여부를 문서화

## 7. Empty State

빈 리스트, 결과 없음, 에러 상태가 프로젝트 곳곳에서 반복됩니다. 이건 기획자와 디자이너가 특히 자주 확인하는 화면이라 디자인시스템 문서화 효과가 큽니다.

대표 구현:

- `NudgeEAPHomePage/src/app/(primary)/components/EmptyResult.tsx`
- `NudgeEAPHomePage/src/app/(primary)/components/EmptyContent.tsx`
- `NudgeEAPWebView/src/app.component/none/NoneEmpty.tsx`
- `NudgeEAPWebView/src/app.component/none/NoneList.tsx`
- `NudgeEAPWebView/src/app.component/none/NoneError.tsx`

추천:

- `EmptyState` 하나로 icon/title/description/action slot 구성
- `empty`, `search-empty`, `error`, `login-required` 같은 상태는 variant보다 recipe로 분리

## 8. Counselor Card

카드 패턴 자체는 반복되지만, 현재는 도메인 의존성이 큽니다. 디자인시스템에 바로 올리기보다는 카드 프리미티브와 레시피로 분리하는 접근이 더 안전합니다.

대표 구현:

- `NudgeEAPHomePage/src/app/(primary)/components/counselor/CounselorCard.tsx`
- `NudgeEAPWebView/src/app.feature/counseling/component/CounselorCard.tsx`

추천:

- `Card`, `CardHeader`, `CardBody`, `CardFooter` 같은 프리미티브부터 정리
- counselor-specific badge, metadata, CTA 조합은 별도 recipe로 유지

## 9. Filter Chip / Tag

필터, 카테고리, 상태 라벨이 섞여 있는 상태입니다. 지금의 `Badge`와 역할이 다른 interactive chip 계열을 분리할지 검토할 만합니다.

대표 구현:

- `NudgeEAPWebView/src/app.component/chip/FilterChip.tsx`
- `NudgeEAPWebView/src/app.feature/coupon/component/CouponBadge.tsx`

추천:

- 읽기 전용 `Badge`와 선택 가능한 `Chip`의 역할 분리
- selected/unselected/disabled 상태를 명확히 문서화

## 10. Verification Row

이 패턴은 컴포넌트 하나로 뽑기보다, `Input + Button + Helper` 조합을 recipe 문서로 먼저 정의하는 게 더 적합해 보입니다.

대표 구현:

- `NudgeEAPHomePage/src/app/(primary)/sign-up/components/SignUpPhoneInput.tsx`
- `NudgeEAPHomePage/src/app/(primary)/sign-up/components/SignUpSMSCodeInput.tsx`
- `NudgeEAPHomePage/src/app/(primary)/find-id-pw/components/FormPhoneVerifyInput.tsx`
- `NudgeEAPHomePage/src/app/(primary)/find-id-pw/components/FormCodeInput.tsx`

추천:

- `Input`의 정식 variant보다는 “인증 입력 패턴” 문서와 Storybook recipe로 관리
- 추후 반복이 더 커지면 `FieldActionRow` 같은 layout primitive를 분리

## 추천 추출 순서

1. Search Input / Search Bar
2. Tabs / Sliding Tabs
3. Bottom Sheet
4. Select / Filter Select
5. Checkbox / Radio / Choice Controls
6. Toast
7. Empty State
8. Filter Chip / Tag
9. Counselor Card 계열 프리미티브

## 바로 실행해볼 만한 작업

- Track A: 기존 `Input`, `Modal`, `Badge`를 확장해서 Search Input, Bottom Sheet, Chip 후보를 먼저 줄이기
- Track B: 새 공통 컴포넌트로 `Tabs`, `Select`, `Toast`, `EmptyState`를 설계하기
- Track C: 각 후보마다 실제 사용 화면 링크와 Figma 링크를 `componentInventory.json`에 연결하기

이 문서는 구현 후보를 빠르게 정리한 버전입니다. 실제 승격 여부는 재사용 빈도, 접근성 요구, Figma 패턴 안정화 정도를 함께 보고 결정하는 것을 권장합니다.
