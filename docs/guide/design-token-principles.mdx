# 디자인 토큰 원칙

디자인 토큰은 단순한 값 모음이 아닙니다.  
디자인 토큰은 디자이너와 개발자가 같은 대상을 같은 이름으로 말하게 해주는 공통 언어입니다.

잘 정의된 토큰은 다음 일을 합니다.

- 시각 기준을 일관되게 유지한다
- 상태별 규칙을 명확하게 만든다
- 디자이너와 개발자 사이의 불필요한 질문을 줄인다
- "디자인이 나와야 개발할 수 있고, 개발이 되어야 상태를 볼 수 있다"는 아이러니를 줄인다
- 한 화면의 수정이 아니라 시스템 단위의 변경을 가능하게 만든다

---

## 1. 토큰은 값이 아니라 의미다

좋은 토큰은 단순히 `#2B96ED` 같은 값을 이름만 붙여 저장한 것이 아니라, 그 값이 어떤 역할을 하는지 설명할 수 있어야 합니다.

예

- 좋은 예: `semantic.primary.main`
- 좋은 예: `semantic.text.disabled`
- 좋은 예: `spacing.16`
- 아쉬운 예: `blue500`
- 아쉬운 예: `cardGray2`

값 자체보다 역할이 먼저 와야, 디자이너와 개발자가 "이 값이 왜 존재하는가"를 함께 이해할 수 있습니다.

---

## 2. 토큰은 디자이너-개발자 공통 언어다

토큰이 잘 정의되어 있으면 대화가 이렇게 바뀝니다.

- "이 버튼 파란색 조금 더 진하게 해주세요"  
  → "primary hover를 조정할까요?"

- "비활성 텍스트가 좀 흐려 보여요"  
  → "text disabled 토큰을 조정할까요?"

- "이 화면만 예외적으로 18px 주세요"  
  → "이게 시스템 spacing인지, 지면 예외인지 먼저 구분할까요?"

좋은 토큰은 디자이너에게는 시각 규칙의 단위이고, 개발자에게는 구현 계약의 단위입니다.

둘 다 토큰을 봐야 일이 빨라집니다.

- 개발은 디자인의 의도를 알아야 정확히 만든다
- 디자인은 개발된 상태와 상호작용을 알아야 올바른 토큰을 만든다

토큰은 이 상호 의존성을 없애지는 못하지만, 훨씬 덜 고통스럽게 만듭니다.

---

## 3. 시각과 상태를 함께 설명해야 한다

디자인 시스템에서 토큰은 정적인 색표만 정의하면 충분하지 않습니다.  
상태별 규칙까지 설명할 수 있어야 합니다.

예를 들면 이런 구분이 필요합니다.

- 기본 텍스트
- 보조 텍스트
- 비활성 텍스트
- 에러 텍스트
- 기본 배경
- hover 배경
- disabled 배경
- focus border
- success background

즉 토큰은 "무슨 색인가"보다 "언제 쓰는가"가 더 중요합니다.

그래서 가능하면 아래 순서를 따르는 편이 좋습니다.

1. 원시 값 정의  
   예: blue 500, neutral 100, red 50
2. 의미 토큰 정의  
   예: primary main, text default, error bg
3. 컴포넌트에서 의미 토큰 사용

컴포넌트는 raw token보다 semantic token을 우선 사용해야 상태 표현이 일관됩니다.

---

## 4. 불필요한 토큰은 만들지 않는다

토큰은 많을수록 좋은 게 아닙니다.  
오히려 의미 없는 토큰이 늘어나면 시스템이 흐려집니다.

대표적인 안 좋은 예:

- `floating-button-color`
- `homepage-card-title-color`
- `modal-blue-text`

이런 값은 정말 토큰이 아니라, 특정 컴포넌트 구현을 고정한 이름일 가능성이 큽니다.

질문은 항상 이거예요.

> 이 값은 여러 곳에서 재사용되는 시스템 규칙인가, 아니면 특정 화면의 구현값인가?

아래 기준으로 나누면 편합니다.

### 토큰으로 정의해도 좋은 경우

- 여러 컴포넌트에서 반복된다
- 상태나 의미를 설명할 수 있다
- 브랜드/시스템 레벨에서 바뀔 가능성이 있다
- 디자이너와 개발자가 같은 이름으로 자주 이야기하게 된다

### 토큰으로 정의하지 않는 편이 좋은 경우

- 특정 컴포넌트 하나에만 묶여 있다
- 특정 지면의 연출 값이다
- 도메인 맥락이 강하다
- 이름을 붙여도 시스템 의미가 거의 없다

플로팅 버튼의 색이 정말로 앱 전반의 "floating action" 규칙이라면 토큰이 될 수 있습니다.  
하지만 그냥 어떤 한 화면에서만 쓰이는 버튼 색이라면 토큰보다 컴포넌트 스타일에 가까운 값일 가능성이 큽니다.

---

## 5. 토큰은 재사용 단위여야 한다

좋은 토큰은 한 번 정의하고 여러 군데에서 같은 의미로 재사용할 수 있어야 합니다.

예

- `spacing.16`: 여러 컴포넌트의 기본 좌우 여백
- `radius.md`: 입력창, 카드, 드롭다운의 기본 radius
- `semantic.border.focus`: input, select, search field의 focus border

이렇게 되면 한 값이 바뀔 때 "이 시스템의 기본 규칙이 바뀌는 것"이라는 설명이 가능합니다.

반대로 한 군데밖에 못 쓰는 값은 토큰이 아니라 그냥 로컬 스타일일 수 있습니다.

---

## 6. 토큰은 예외를 줄이는 방향으로 설계한다

토큰이 많아지는 대표 이유는 "이번 화면만 조금 다르게"가 계속 누적되기 때문입니다.

그럴수록 먼저 확인해야 합니다.

- 이 예외가 진짜 시스템 규칙이 될 가능성이 있는가
- 아니면 이번 지면에서만 필요한 연출인가
- 기존 토큰 조합으로 해결 가능한가
- 새로운 토큰을 추가하면 오히려 혼란이 커지지 않는가

토큰은 예외를 저장하는 창고가 아니라, 예외를 줄이기 위한 기준이어야 합니다.

---

## 7. 토큰은 구현보다 먼저 정의하되, 구현으로 검증한다

토큰은 디자인에서 먼저 출발할 수 있지만, 실제로 좋은 토큰인지는 구현을 거쳐야 드러납니다.

예를 들면:

- 디자이너는 `disabled text`를 정의했는데
- 개발 과정에서 `disabled icon`, `disabled border`, `disabled background`도 같이 필요하다는 걸 알 수 있습니다

반대로 개발 중에 "hover용 색을 따로 만들자"는 요구가 나와도, 디자인에서 봤을 때 시스템 의미가 없으면 토큰으로 올리지 않는 편이 맞습니다.

좋은 토큰은 디자인과 개발이 서로를 확인하면서 다듬어집니다.

---

## 8. 추천하는 토큰 레이어

NDS에서는 토큰을 아래처럼 생각하는 편이 자연스럽습니다.

### 1. Primitive Tokens

가장 기초가 되는 값입니다.

- color palette
- spacing scale
- radius scale
- typography scale
- z-index
- motion duration

예

- `colors.blue[500]`
- `spacing[16]`
- `radius.md`

### 2. Semantic Tokens

실제 역할을 설명하는 값입니다.

- `semantic.text.default`
- `semantic.text.disabled`
- `semantic.primary.main`
- `semantic.error.bg`
- `semantic.border.focus`

디자인 시스템 컴포넌트는 가능하면 이 레이어를 직접 쓰는 편이 좋습니다.

### 3. Component Tokens

정말 공통 컴포넌트 레벨에서 반복되고, 외부 오버라이드 가치가 있을 때만 둡니다.

예

- `--nds-button-radius`
- `--nds-input-height`
- `--nds-select-dropdown-max-height`

이 레이어는 유용하지만, 너무 많아지면 컴포넌트 구현 세부를 그대로 공개하는 셈이 되므로 절제해서 사용합니다.

---

## 9. 좋은 토큰 이름의 특징

좋은 토큰 이름은 아래 조건을 만족할수록 좋습니다.

- 의미가 있다
- 상태를 설명할 수 있다
- 특정 화면 이름이 들어가지 않는다
- 특정 컴포넌트 이름이 불필요하게 들어가지 않는다
- raw value보다 역할이 먼저 보인다

예

- 좋음: `semantic.text.subtle`
- 좋음: `semantic.success.bg`
- 좋음: `spacing.24`
- 아쉬움: `homepage-title-gray`
- 아쉬움: `floating-btn-blue`
- 아쉬움: `popup-main-text-color`

---

## 10. 토큰을 추가하기 전에 묻는 질문

새 토큰을 만들기 전에 아래 질문을 먼저 봅니다.

1. 이 값은 반복되는가?
2. 이 값은 상태나 역할을 설명하는가?
3. 여러 화면/컴포넌트에서 같은 의미로 쓸 수 있는가?
4. 기존 토큰 조합으로 해결할 수 없는가?
5. 이 이름을 디자이너와 개발자가 함께 계속 쓸 것 같은가?

여기서 여러 개가 `아니오`라면, 토큰 추가보다 로컬 스타일이나 recipe 레벨 해결이 더 맞을 수 있습니다.

---

## 11. NDS에서 지향하는 토큰 태도

정리하면 NudgeEAP Design System의 토큰은 이렇게 가는 편이 좋습니다.

- 토큰은 값 목록이 아니라 시스템 언어다
- 토큰은 시각과 상태를 함께 설명해야 한다
- 토큰은 의미 중심으로 이름 붙인다
- 토큰은 재사용 단위일 때만 추가한다
- 토큰은 예외를 저장하지 않는다
- 토큰은 디자인과 개발이 함께 검증한다

좋은 토큰은 디자이너와 개발자 사이의 커뮤니케이션을 줄여주면서도, 필요한 대화는 더 정확하게 만들어줍니다.

실무에서 새 토큰 추가나 수정 요청을 검토할 때는 [토큰 리뷰 체크리스트](/guide/token-review-checklist)를 함께 참고합니다.
