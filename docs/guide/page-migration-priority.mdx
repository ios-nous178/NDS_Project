# 페이지 교체 우선순위

디자인 시스템을 실제 소비 프로젝트에 적용할 때는 "가장 눈에 띄는 화면"보다 "가장 학습 효율이 높은 화면"부터 교체하는 편이 안정적입니다.

이 문서는 NudgeEAP HomePage 같은 실제 서비스 코드에 디자인 시스템을 처음 적용할 때, 어떤 지면부터 교체하는 것이 좋은지 판단하기 위한 가이드입니다.

---

## 기본 원칙

첫 교체 대상은 아래 조건을 많이 만족할수록 좋습니다.

- 이미 디자인 시스템 컴포넌트로 커버 가능한 UI가 많다
- 도메인 로직은 비교적 단순하다
- SEO나 전환 지표에 대한 직접 영향이 작다
- 기존 화면과 새 화면을 나란히 비교하기 쉽다
- Flat API로 먼저 바꾸고, 필요한 부분만 Compound API로 확장할 수 있다

반대로 아래 조건이 강한 화면은 첫 대상에서 미루는 편이 좋습니다.

- 홈 첫 화면처럼 비즈니스 임팩트가 아주 크다
- 예약, 결제, 완료 같은 핵심 전환 플로우다
- 모달, 바텀시트, 포커스 트랩, 스크롤 락 의존이 많다
- 도메인 카드와 비동기 상태가 복잡하게 엮여 있다

---

## 추천 1순위

### 검색/리스트형 지면

예시

- 상담사 목록
- 콘텐츠 목록
- 검색 결과 화면
- 카테고리 탭 + 검색 + 필터 + 리스트 + empty state 조합 화면

이 지면을 먼저 추천하는 이유

- 현재 디자인 시스템 컴포넌트와 궁합이 좋음  
  `Tabs`, `SearchInput`, `Select`, `Badge`, `EmptyState`, `Toast`
- 스타일 통일 효과가 바로 보임
- 사용자가 체감하는 변화는 크지만, 비즈니스 리스크는 상대적으로 낮음
- 기존 구현과 parity 비교가 쉬움
- 디자인 시스템의 Flat API를 먼저 검증하기 좋음

---

## 추천 2순위

### 탭 중심 정보 탐색 지면

예시

- 카테고리 탭 전환 화면
- 주소/센터 검색 전환 화면
- 정보 섹션을 탭으로 나누는 화면

이 지면을 두 번째로 추천하는 이유

- `Tabs`의 실제 사용 맥락을 검증하기 좋음
- Flat API와 Compound API를 상황별로 나눠 쓰기 좋음
- WebView와 HomePage 사이 UI parity를 비교하기 쉬움

단, 아래 항목은 같이 봐야 합니다.

- 긴 라벨 overflow
- active tab indicator
- 키보드 접근성
- 패널 없는 탭 리스트 사용 여부

---

## 추천 3순위

### 가벼운 입력/문의 지면

예시

- 간단한 검색/문의 입력 화면
- 필터 입력 화면
- 에러/헬퍼 텍스트가 중요한 폼 일부

이 지면이 좋은 이유

- `Input`, `SearchInput`, `FieldActionRow`, `Button` 조합을 점검하기 좋음
- form contract를 디자인 시스템 기준으로 통일하기 쉬움
- 레이아웃보다 상태 계약을 검증하기 좋음

주의할 점

- 복잡한 검증 로직이 붙은 폼 전체를 한 번에 바꾸지 않는다
- 먼저 재사용 가능한 입력 블록 단위로 쪼개서 교체한다

---

## 첫 교체 대상으로 비추천

### 메인 홈 첫 화면

비추천 이유

- SEO와 전환에 미치는 영향이 큼
- 비주얼 민감도가 높음
- 작은 스타일 차이도 체감되기 쉬움

### 예약/결제/완료 플로우

비추천 이유

- 비즈니스 리스크가 큼
- QA 범위가 넓어짐
- 단순 UI 교체가 아니라 플로우 검증까지 필요해짐

### 오버레이 의존 지면

예시

- 모달 중심 화면
- 바텀시트 중심 화면
- 다단계 팝업 흐름

비추천 이유

- `Modal`, `Popup`, `BottomSheet`는 여전히 web-specific behavior 분리를 더 진행할 여지가 있음
- 포커스 트랩, body scroll lock, portal 위치 같은 항목을 같이 검증해야 함

## 실제 선정 기준표

페이지를 고를 때 아래 표로 점수화하면 편합니다.

| 기준 | 질문 | 권장 방향 |
|---|---|---|
| 컴포넌트 적합도 | 현재 DS 컴포넌트로 60% 이상 커버 가능한가 | 높을수록 우선 |
| 도메인 복잡도 | 데이터/분기/예외 처리가 복잡한가 | 낮을수록 우선 |
| 전환 민감도 | 실패 시 비즈니스 영향이 큰가 | 낮을수록 우선 |
| 비교 가능성 | 기존 UI와 새 UI를 나란히 비교하기 쉬운가 | 높을수록 우선 |
| QA 용이성 | visual QA와 parity 체크가 쉬운가 | 높을수록 우선 |
| 구조 학습 가치 | DS의 핵심 contract를 실제로 검증하기 좋은가 | 높을수록 우선 |

---

## 권장 진행 순서

1. 검색/리스트형 지면 하나를 선정한다.
2. 기존 화면을 기준으로 사용 중인 UI 패턴을 분해한다.
3. 디자인 시스템 컴포넌트로 바로 대체 가능한 부분과 recipe로 남겨야 할 부분을 나눈다.
4. Flat API로 먼저 치환한다.
5. 실제 커스터마이징이 필요한 부분만 Compound API로 내린다.
6. Storybook parity story와 실지면 비교 스크린샷을 함께 본다.
7. 한 화면에서 패턴이 안정되면 다음 지면으로 확장한다.

핵심은 "가장 큰 화면"부터가 아니라 "가장 반복 가능한 패턴이 많은 화면"부터 시작하는 것입니다.

---

## 지금 기준 추천

현재 디자인 시스템 상태를 기준으로 하면, 첫 적용 지면은 아래 순서가 가장 무난합니다.

1. 검색/리스트형 페이지
2. 탭 중심 정보 탐색 페이지
3. 가벼운 입력/문의 페이지
4. 그 다음에 오버레이가 많은 페이지

한 줄로 정리하면 이렇습니다.

> 도메인 복잡도는 낮고, 디자인 시스템 컴포넌트 사용 밀도는 높은 페이지부터 교체한다.
