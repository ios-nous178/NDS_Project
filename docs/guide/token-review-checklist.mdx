# 토큰 리뷰 체크리스트

이 문서는 새 디자인 토큰을 추가하거나 기존 토큰을 수정할 때,  
"이걸 정말 토큰으로 만들어야 하는가?"를 빠르게 판단하기 위한 실무용 체크리스트입니다.

원칙 설명은 [디자인 토큰 원칙](/guide/design-token-principles)을 참고하고, 이 문서는 실제 리뷰와 의사결정에 바로 쓰는 문서로 봅니다.

---

## 1. 먼저 묻는 질문

새 토큰 제안이 들어오면 가장 먼저 아래를 확인합니다.

1. 이 값은 반복되는가?
2. 이 값은 역할이나 상태를 설명하는가?
3. 여러 화면이나 컴포넌트에서 같은 의미로 재사용 가능한가?
4. 기존 토큰 조합으로 해결할 수 없는가?
5. 디자이너와 개발자가 같은 이름으로 계속 부를 가능성이 있는가?

여기서 여러 개가 `아니오`면, 새 토큰보다 로컬 스타일이나 recipe로 푸는 편이 더 좋을 수 있습니다.

---

## 2. 결정 트리

아래 순서로 판단하면 대부분의 케이스를 정리할 수 있습니다.

### 1단계. 이 값이 시스템 규칙인가?

- 예  
  다음 단계로 진행
- 아니오  
  토큰보다 로컬 스타일 또는 지면 전용 구현에 가깝다

예

- 시스템 규칙에 가까움: disabled text color, focus border, spacing 16
- 시스템 규칙과 거리가 멂: 특정 프로모션 배너의 배경색, 한 화면의 장식용 그림자

### 2단계. 이 값이 의미를 설명하는가?

- 예  
  semantic token 후보
- 아니오  
  primitive token 또는 토큰 아님

예

- 의미 있음: `semantic.text.disabled`
- 의미 약함: `homepage-blue-text`

### 3단계. 이 값이 컴포넌트 레벨 확장 포인트인가?

- 예  
  component token 또는 CSS variable 후보
- 아니오  
  semantic token 또는 로컬 구현으로 유지

예

- component token 후보: `--nds-input-height`, `--nds-button-radius`
- component token으로 보기 어려움: `--nds-home-hero-badge-shadow`

---

## 3. 이건 토큰으로 올릴 만하다

아래 중 여러 개에 해당하면 토큰으로 정의할 가능성이 높습니다.

- 여러 컴포넌트에서 반복 사용된다
- 상태 차이를 표현한다
- 브랜드/시스템 변경 시 함께 조정될 가능성이 있다
- 문서화했을 때 팀 전체가 같은 의미로 이해할 수 있다
- 특정 화면을 넘어서도 설명이 된다

예

- text default / subtle / disabled
- border default / focus / error
- success / error / caution background
- spacing scale
- radius scale
- elevation scale
- motion duration / easing

---

## 4. 이건 토큰으로 올리지 않는 편이 낫다

아래 중 여러 개에 해당하면 토큰보다 로컬 스타일일 가능성이 높습니다.

- 한 지면에서만 쓰인다
- 도메인 맥락이 아주 강하다
- 특정 마케팅 연출을 위한 값이다
- 특정 컴포넌트 하나에만 의미가 있다
- 이름을 붙여도 시스템 의미가 빈약하다

예

- `floating-button-color`
- `homepage-card-title-color`
- `promo-banner-shadow`
- `modal-blue-text`

이런 값은 일단 토큰보다 컴포넌트 스타일이나 recipe 레벨에서 해결하는 편이 더 깔끔합니다.

---

## 5. 어떤 레이어에 둘 것인가

토큰으로 보기로 했다면, 다음엔 어느 레이어에 둘지 판단합니다.

### Primitive Token

값의 기본 단위입니다.

예

- `colors.blue[500]`
- `spacing[16]`
- `radius.md`
- `motion.duration.fast`

이런 경우에 적합합니다.

- 디자인 팔레트나 기본 스케일을 정의할 때
- 아직 역할보다 값 자체를 정리하는 단계일 때

### Semantic Token

역할과 상태를 설명하는 단위입니다.

예

- `semantic.primary.main`
- `semantic.text.disabled`
- `semantic.border.focus`
- `semantic.error.bg`

이런 경우에 적합합니다.

- 컴포넌트가 직접 참조할 값일 때
- 상태와 의미가 중요할 때
- 디자이너와 개발자가 공통 언어로 써야 할 때

### Component Token

공통 컴포넌트 차원에서 반복되고, 외부 조정 가치가 있을 때만 둡니다.

예

- `--nds-button-radius`
- `--nds-input-height`
- `--nds-select-dropdown-max-height`

이런 경우에 적합합니다.

- 컴포넌트의 확장 포인트가 분명할 때
- 소비 프로젝트에서 바꿀 가능성이 높을 때

주의할 점

- component token은 편하지만 너무 늘리면 컴포넌트 내부 구현을 그대로 노출하게 됩니다
- "조절 가능하다"와 "토큰이어야 한다"는 같은 말이 아닙니다

---

## 6. 수정 요청이 들어왔을 때 체크할 것

기존 토큰 변경은 새 토큰 추가보다 더 조심해야 합니다.

아래를 먼저 봅니다.

- 변경 범위가 어디까지인가
- 어떤 컴포넌트들이 영향받는가
- primitive 수정인지 semantic 수정인지
- 디자인 의도 변경인지 단일 화면 보정인지
- 시각 QA와 컴포넌트 QA에서 무엇을 다시 봐야 하는가

예

- `primary.main` 변경  
  버튼, 링크, 포커스, 선택 상태까지 광범위하게 영향 가능
- `spacing.16` 변경  
  입력창, 카드, 모달, 리스트 padding에 동시 영향 가능

즉 토큰 수정은 "한 값 수정"이 아니라 "시스템 규칙 수정"일 수 있습니다.

---

## 7. 리뷰 코멘트 예시

실제 리뷰에서는 아래처럼 말하면 좋습니다.

### 토큰 추가가 적절한 경우

- "이 값은 여러 컴포넌트의 focus 상태에서 반복되니 semantic token으로 올리는 편이 좋겠습니다."
- "이건 input과 select가 공통으로 쓰는 높이라 component token으로 노출할 가치가 있습니다."

### 토큰 추가가 과한 경우

- "이 값은 현재 홈 한 화면에서만 쓰여서 토큰보다 로컬 스타일에 가까워 보입니다."
- "이름만 보면 시스템 의미가 약해서 semantic token으로 올리기엔 아직 이른 것 같습니다."
- "기존 semantic token 조합으로 해결 가능하니 새 토큰을 추가하지 않는 편이 좋겠습니다."

---

## 8. 최종 체크리스트

토큰 제안 하나를 리뷰할 때 아래 항목을 빠르게 체크합니다.

- 반복 사용되는 값인가
- 의미나 상태를 설명하는가
- 특정 지면 이름에 묶여 있지 않은가
- 기존 토큰 조합으로 해결 불가능한가
- 시스템 규칙으로 문서화할 가치가 있는가
- primitive / semantic / component 중 어디에 둘지 설명 가능한가
- 변경 시 영향 범위를 대략 예측할 수 있는가

이 중 설명이 잘 안 되는 항목이 많으면, 토큰보다 로컬 구현일 가능성이 큽니다.

---

## 9. NDS 기준 추천 태도

NudgeEAP Design System에서는 아래 태도를 추천합니다.

- 토큰은 적을수록 좋다가 아니라, 의미 있을수록 좋다
- 시스템 규칙을 드러내는 토큰은 아끼지 않는다
- 특정 화면 예외를 토큰으로 흡수하지 않는다
- semantic token을 중심으로 대화한다
- component token은 정말 필요한 확장 포인트에만 둔다

한 줄로 말하면 이렇습니다.

> 토큰은 구현을 설명하는 값이 아니라, 시스템 규칙을 설명하는 이름이어야 한다.
