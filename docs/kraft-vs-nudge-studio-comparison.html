Kraft vs Nudge Studio 구조 비교

당근 글의 핵심 구조를 기준으로, 현재 Nudge Studio 데스크톱 앱의 생성 전 구조화, 로컬 에이전트, DS 검증, 메모리 레이어를 나란히 비교한 화면입니다.

기준 글: 당근 테크 블로그, “프롬프트 한 줄로 화면이 나오는 시대, ‘당근스러운 화면’을 만드는 법”
로컬 기준: apps/desktop, packages/mcp, packages/mockup-core

Kraft 의사결정 자동화 루프

1
사용자 요청 “거래 후기 작성 화면 만들어줘” 같은 자연어 입력.
2
Clarify 모호한 상태, 입력 방식, CTA 조건 등을 먼저 질문해 요구사항을 구체화.
3
Memory Read + Cases 과거 결정과 유사 화면 레퍼런스를 찾아 이번 설계에 반영.
4
DesignSpec JSON 구조, 컴포넌트, 토큰, 상태, 디자인 결정 근거를 코드와 분리해 저장.
5
Generate 확정된 DesignSpec을 기준으로 실제 React 코드를 생성.
6
Verify SEED 컴포넌트, 토큰, 레이아웃, UX 상태 처리 등을 scorer로 평가.
7
Memory Write 좋은 결정과 교정 이력을 다음 세션에서 재사용할 원칙으로 누적.

Nudge Studio 현재 데스크톱 앱 흐름

1
Intake Modal 브랜드, 표면, 플랫폼, 화면명, PRD, Figma/스크린샷, 첨부 문서를 입력.
2
워크스페이스 생성 brief.md, references.md, CLAUDE.md, AGENTS.md, nudge.brand를 생성.
3
로컬 에이전트 실행 Claude/Codex를 작업 폴더 cwd에서 실행하고 세션/터미널 로그를 저장.
4
nudge-ds MCP 조회 get_guide, find_component, find_token, find_icon.
5
HTML 목업 생성 <nds-*> 커스텀 엘리먼트와 시멘틱 토큰으로 index.html 작성.
6
검증 + 빌드 validate_html_mockup, build_singlefile_html로 위반과 사용량을 확인.
7
기록 세션 로그, 구조화 메시지, 피드백, 앱 이벤트를 로컬에 축적.

핵심 비교

비교 축 Kraft Nudge Studio 현재 상태 판단
생성 전 구조화 Clarify와 DesignSpec으로 요구사항을 코드 전 단계에서 정리. Intake가 입력값을 받아 brief/references/AGENTS 파일로 정리. 강함
로컬 컨텍스트 CLI가 프로젝트 폴더, 팀 문서, 기존 코드를 읽음. Electron이 작업 폴더를 만들고 에이전트를 해당 cwd에서 실행. 강함
DS 지식 주입 SEED 문서 MCP와 스킬 파일을 필요한 시점에 로드. nudge-ds MCP + 시스템 프롬프트로 가이드/컴포넌트/토큰 조회를 강제. 강함
중간 표현 DesignSpec JSON이 설계 의도를 코드와 분리. brief.md는 있으나 컴포넌트/토큰/상태/근거를 담는 JSON spec은 없음. 보강 필요
검증 정적 scorer + LLM scorer로 품질을 평가하고 수정 루프를 만듦. HTML validator와 build validator가 DS 위반을 잡고 리포트함. 기반 있음
크로스세션 메모리 decision-log와 principles를 쌓아 다음 세션에 자동 주입. 세션/피드백 로그는 있지만 디자인 결정이 원칙으로 승격되지는 않음. 핵심 갭

에이전트 아키텍처와 스킬 시스템 비교

Kraft가 명시적으로 가진 것

Plan / Orchestra 모드 검토 모드는 설계와 승인까지만, 실행 모드는 설계 → 검증 → 코드 생성까지 end-to-end로 처리. 코드 생성 권한을 모드별로 분리한다.
설계 도구와 코딩 도구 분리 clarify, case lookup, spec 저장/검증 같은 설계 도구와 runCodingAgent를 분리해 “무엇을 만들지”와 “어떻게 고칠지”를 나눈다.
DesignSpec 중간 표현 프롬프트에서 바로 코드로 가지 않고 JSON spec을 거쳐 설계 의도와 근거를 남긴다.
Skill 시스템 spacing, radius, typography, screen-patterns, writing-guide, design-principles, eval-self-check를 마크다운 모듈로 분리하고 필요한 순간에만 로드한다.
Eval / Memory 루프 생성 결과를 scorer로 채점하고, 결정 로그를 다음 세션의 원칙으로 승격한다.

Nudge Studio 현재 구조

인테이크는 강함 Intake Modal이 브랜드/표면/플랫폼/PRD/레퍼런스를 받아 brief.md와 AGENTS.md를 생성한다. 생성 전 게이트는 이미 꽤 명확하다.
MCP 가이드는 있음 get_guide, find_component, find_token, find_icon, validate_html_mockup 등은 디자인 시스템 지식과 검증을 도구화한다.
하지만 Agent Harness가 약함 현재는 Claude/Codex CLI를 띄우는 실행 어댑터에 가깝다. Plan/Orchestra 같은 역할 분리, 의도 라우팅, 설계 전용 도구 권한 경계가 제품 구조로 명시되어 있지 않다.
Skill Loader가 없음 MCP guide는 있지만 “필요한 시점에 필요한 규칙만 읽는” 스킬 선택기가 없다. 시스템 프롬프트와 큰 guide에 의존하면 토큰 낭비와 노이즈가 생긴다.
Spec / Memory가 없음 brief와 세션 로그는 남지만, 디자인 결정이 spec과 memory로 축적되어 다음 세션에 자동 주입되는 구조는 아직 없다.

Eval과 크로스세션 메모리 비교

Kraft Eval 구조

코드 기반 scorer

  • color-tokens: 하드코딩 hex 대신 시맨틱 토큰 사용
  • typography: 텍스트 크기 단계의 논리성
  • layout-structure: 레이아웃 구조 규칙 준수
  • spacing-rules: spacing 토큰의 맥락 적합성
  • component-compliance: 컴포넌트 props 정확성
  • icon-usage: 아이콘 라이브러리 규칙 준수
  • animation-stability: 애니메이션 안정성

LLM 기반 scorer

  • ux-patterns: 화면 유형에 맞는 UX 패턴
  • interaction-quality: 인터랙션 흐름 품질
  • flow-patterns: 화면 간 흐름 적합성
  • form-patterns: 에러/로딩/빈 상태 처리 충실도

Nudge Studio 현재 Eval 상태

이미 있는 것 validate_html_mockup과 build_singlefile_html이 토큰, spacing, native element, icon, pattern 위반을 잡는다. 정적 검증 기반은 있다.
없는 것: 점수화 모델 위반 0/건수 중심이지, color/typography/layout/icon/UX/form 품질을 scorer별 점수로 산출하고 합산하는 평가판은 아직 없다.
없는 것: 자동 수정 루프 낮은 점수 → 원인 요약 → 재생성/수정 → 재평가를 에이전트 하네스가 관리하는 구조가 아직 명시되어 있지 않다.
없는 것: LLM 정성 평가 UX 패턴, 플로우, 인터랙션, 폼 상태 같은 정성 품질을 별도 LLM scorer로 보는 단계가 없다.

크로스세션 디자인 메모리

1. designDecisions 생성/수정 때 내린 결정과 근거를 spec에 기록.
2. decision-log.jsonl 세션 단위로 결정 로그를 누적.
3. principles.json 반복 패턴을 브랜드/표면/도메인별 원칙으로 승격.
4. Memory Read 다음 작업 시작 시 관련 원칙을 brief/spec에 자동 주입.

현재 Nudge Studio는 세션 로그와 피드백 로그는 있지만, 이 4단계를 디자인 결정 데이터로 연결하지는 않는다. Kraft 대비 가장 큰 제품 진화 포인트다.

지금 이미 좋은 점

단순 프롬프트 입력이 아니라 Intake에서 화면 제작 조건을 먼저 수집한다. 이 구조는 Kraft의 Clarify 단계와 같은 문제를 이미 어느 정도 해결한다.

가장 큰 빈칸

design-spec.json이 없다. 그래서 설계 의도, 컴포넌트 선택 이유, 상태 처리 원칙이 코드와 분리된 데이터로 남지 않는다.

다음 도약

검증 결과와 사용자 피드백을 decision log로 저장하고, 반복되는 결정을 브랜드/표면별 원칙으로 승격해 다음 작업에 자동 주입한다.

추천 목표 구조

Nudge Studio가 Kraft와 같은 수준의 “학습하는 디자인 시스템 에이전트”가 되려면 아래 순서가 가장 직접적입니다. 핵심은 CLI 실행기를 감싸는 Agent Harness와 Skill Loader를 별도 레이어로 올리고, 검증 결과를 점수화한 뒤 디자인 메모리로 되돌리는 것입니다.

1. Intake 브랜드/표면/플랫폼/PRD/레퍼런스 수집
2. Agent Harness Plan/Orchestra 모드와 의도 라우팅
3. Skill Loader spacing/radius/typography/pattern 등 필요한 규칙만 로드
4. design-spec.json 구조, 상태, 컴포넌트, 토큰, 결정 근거 저장
5. Generate spec + MCP 기준으로 HTML 목업 생성
6. Eval Scorer 정적 scorer + LLM scorer로 품질 점수화
7. Auto Fix Loop 낮은 점수 원인 분석 → 수정 → 재평가
8. Decision Log designDecisions를 jsonl로 축적
9. Memory Read principles.json을 다음 작업에 자동 주입
